# 全文检索服务设计文档

## 一、 技术选型与机制调研

### 1.1 MongoDB Text Index（文本索引）机制
MongoDB 提供了内置的 **Text Index**（文本索引）以支持对字符串内容的全文搜索。

* **单一索引限制**：每个 MongoDB 集合（Collection）最多只能拥有**一个**文本索引。但该索引可以是一个包含多个字段的**复合文本索引（Compound Text Index）**。
* **分词器（Tokenizer）限制**：
  * **空格分词**：MongoDB 社区版默认使用拉丁语系的分词机制（基于空格和标点符号分词）。
  * **中文分词局限**：由于中文句子中词语之间没有空格，MongoDB 默认的分词器会将整句中文或大部分连续字符视为一个词。因此，在不使用外部工具或 MongoDB 4.0+ 的 ICU 分词插件（需要在创建索引时指定 `default_language: "none"` 配合特定配置）的前提下，MongoDB 对中文全文索引的匹配精度较差，无法支持智能分词搜索（如输入“物联网”可能搜不出“物联”或“网”）。
* **权重（Weights）配置**：
  * 在复合文本索引中，可以为不同的字段分配不同的权重，以此来决定搜索相关度得分（Relevance Score）。
  * 例如，将 `title` 字段的权重设为 `10`，将 `content` 字段的权重设为 `1`。当关键词在标题中匹配时，其相关性得分会显著高于在正文中匹配，从而使更重要的文档在结果列表中排在前面。
* **相关度排序（textScore）**：
  * 在执行文本查询时，MongoDB 会为每个匹配的文档计算一个投影得分 `$meta: "textScore"`。
  * 开发者可以使用这个得分进行降序排列，实现按“相关度”从高到低呈现结果。

### 1.2 高亮实现方案调研（基于企业高涉密、非云托管背景）
由于企业涉密性要求极高，**无法使用 MongoDB Atlas Search 等任何公有云托管方案**。因此，底层的全文检索必须基于私有化部署（本地/私有云）的 MongoDB 社区版。

在这种完全私有化的受限环境下，高亮显示无法依赖 MongoDB 官方云端原生的 `$search` 聚合高亮。我们只能在以下两种本地化高亮方案中进行权衡：

#### 方案 A：后端（Java 服务端）手动提取片段并进行高亮（推荐）
* **实现原理**：微服务在通过 `TextQuery` 检索出匹配的 MongoDB 原始文档后，在 Java 服务端（Service 层）对数据进行二次处理：
  1. **分词解析**：将用户的输入关键词（如空格分隔的多词）拆分为关键词集合。
  2. **高亮匹配**：在文本中匹配到这些关键词后，用 HTML 标记（如 `<mark>` 或 `<span>`）包裹。
  3. **片段截取（Snippet Extraction）**：根据关键词位置，向上和向下截取固定长度（如前后 100 字符）的文本，形成“高亮摘要片段”，并随 VO 返回给前端。
* **优势**：
  * **节省网络带宽**：避免将几万字的长文档全文传给前端，只需返回包含关键字的片段（Snippet），极大地降低了带宽消耗和网络传输延迟。
  * **多关键词支持精确**：服务端的拆词逻辑和底层的 MongoDB 检索拆词逻辑一致，能够完美实现多关键词（“空格”分隔）分别高亮。
  * **安全性高**：在后端可以通过 XSS 过滤器对内容进行统一转义消毒，再安全地包裹高亮标签，防止前端渲染任意 HTML 时的注入攻击。
* **劣势**：在 Java 服务端做大批量、大文本的正则或多模式字符串替换，会对 CPU 产生一定的计算压力，需要良好的代码实现和缓存优化。

#### 方案 B：前端（Client-Side）基于正则替换进行高亮（当前已有实现）
* **实现原理**：后端服务只负责把数据库中的 `title` 和 `content` 原封不动地通过 JSON 返回给前端。前端 Vue 代码中定义了 `highlightText(text, searchKey)` 方法，利用 `new RegExp` 正则表达式在浏览器端对整个文本进行全局正则替换，并使用 `v-html` 动态渲染。
* **优势**：
  * **实现极度简单**：后端几乎不需要任何高亮代码，数据库和微服务开销最小。
  * **前端动态性高**：可以直接结合前端组件动态调整样式。
* **劣势（对于企业级全文检索是致命的）**：
  * **性能瓶颈**：如果业务文档的正文字符数极大（如几万字的报告或合同），后端必须将完整的正文通过网络传输给前端，导致严重的带宽浪费和慢加载；前端对大文本执行正则替换并重新渲染 DOM，容易造成浏览器卡顿甚至崩溃。
  * **多关键词搜索失效（当前Bug）**：如果用户输入了多个关键词（如 `“物联网 大模型”`），后端全文检索能分别搜出包含这两个词的文档，但前端如果拿整串 `“物联网 大模型”` 作为正则去匹配，将因为无法精确分词而**完全搜不出任何高亮**。
  * **XSS 安全漏洞风险**：前端为了渲染高亮标签，不得不使用 `v-html` 渲染后端返回的文本。如果文档正文中夹带了恶意的 `<script>` 注入，将直接引发 XSS 跨站脚本攻击。这在涉密高、安全要求严格的企业级系统中是绝对不被允许的。

## 二、 数据模型设计

根据任务背景，全文检索服务需要能够存储各类业务数据。我们将通用设计模型与当前项目中的实际数据模型进行对比：

### 2.1 通用全文检索模型（概念设计）
为支持多业务实体检索，设计了一个扁平且通用的文档结构：

| 字段名 | 类型 | 索引类型 | 说明 |
| :--- | :--- | :--- | :--- |
| `id` | String | 主键 | MongoDB 自动生成的 ObjectId |
| `businessType` | String | 单字段索引 | 业务类型（如：`news`、`article`、`product`） |
| `businessId` | String | 复合唯一索引 | 业务数据在原表中的主键 ID |
| `title` | String | 文本索引（权重：10） | 文档标题，支持全文搜索 |
| `content` | String | 文本索引（权重：1） | 文档正文，支持全文搜索 |
| `tags` | List\<String\> | 文本索引（权重：5） | 标签/关键字，支持精确过滤和辅助搜索 |
| `createTime` | Date | 普通索引（降序） | 业务数据的创建时间，用于按时间范围检索 |

### 2.2 实际落地模型（项目已有代码 `MongoNewsDO`）
当前项目的具体实现以“新闻（News）”为例，设计了如下存储结构：

```java
@Document(collection = "cms_news")
@Data
public class MongoNewsDO {
    @Id
    private String id;
    
    private String title;             // 新闻标题
    private List<String> keywords;    // 新闻关键字
    private String content;           // 新闻正文内容
    private LocalDateTime createTime;  // 创建时间
    private Map<String, Object> extra; // 其他自定义动态字段（Schema-less 动态容器）
}
```

> **设计解析**：
> `MongoNewsDO` 通过 `extra` 字段利用了 MongoDB 的 Schema-less（无模式）特性，允许客户端在不同场景下动态上传额外的属性（例如作者、阅读量、状态等），极大地提高了字段的扩展性。

## 三、 REST API 接口设计

全文检索服务遵循 `yudao-cloud` 的标准 RESTful API 规范，约定全局统一返回格式为 `CommonResult`。

### 3.1 核心 API 说明书

#### 1. 创建新闻文档
* **请求路径**: `POST /admin-api/mongo/news/create`
* **接口描述**: 新增一条检索文档。
* **请求参数 (JSON Body)**:
```json
{
  "title": "大模型在物联网领域的创新应用",
  "keywords": ["大模型", "IoT", "人工智能"],
  "content": "近年来，大语言模型技术在千行百业落地...",
  "extra": {
    "author": "张三",
    "views": 250
  }
}
```
* **响应参数 (JSON)**:
```json
{
  "code": 0,
  "data": "66935cc235d94e21a8c3d9a2",
  "msg": ""
}
```

#### 2. 修改新闻文档
* **请求路径**: `PUT /admin-api/mongo/news/update`
* **接口描述**: 更新已有文档的内容与动态字段。
* **请求参数 (JSON Body)**:
```json
{
  "id": "66935cc235d94e21a8c3d9a2",
  "title": "大模型在物联网领域的创新应用(修改版)",
  "keywords": ["大模型", "IoT"],
  "content": "更新后的正文内容...",
  "extra": {
    "author": "李四",
    "views": 251
  }
}
```
* **响应参数 (JSON)**:
```json
{
  "code": 0,
  "data": true,
  "msg": ""
}
```

#### 3. 删除新闻文档
* **请求路径**: `DELETE /admin-api/mongo/news/delete`
* **接口描述**: 根据 ID 删除索引文档。
* **请求参数 (Query String)**: `id=66935cc235d94e21a8c3d9a2`
* **响应参数 (JSON)**:
```json
{
  "code": 0,
  "data": true,
  "msg": ""
}
```

#### 4. 分页检索新闻（支持全文检索与普通条件查询）
* **请求路径**: `GET /admin-api/mongo/news/page`
* **请求参数 (Query String)**:
  * `pageNo` (Integer, 必填): 当前页码
  * `pageSize` (Integer, 必填): 每页大小
  * `searchType` (String, 选填): 检索范围，`all` (全文检索), `title` (仅标题), `content` (仅正文)
  * `searchKey` (String, 选填): 搜索关键词
* **响应参数 (JSON)**:
```json
{
  "code": 0,
  "msg": "",
  "data": {
    "list": [
      {
        "id": "66935cc235d94e21a8c3d9a2",
        "title": "大模型在物联网领域的创新应用",
        "keywords": ["大模型", "IoT"],
        "content": "正文内容...",
        "createTime": "2026-07-14T10:04:17",
        "extra": {
          "author": "李四",
          "views": 251
        }
      }
    ],
    "total": 1
  }
}
```

#### 5. 批量导入新闻
* **请求路径**: `POST /admin-api/mongo/news/batch-import`
* **请求参数 (JSON Body)**: 接收包含多条新闻的数组。
* **响应参数 (JSON)**: `{"code": 0, "data": true, "msg": ""}`

## 五、 项目已有实现检查与差距分析（代码审查报告）

根据对当前分支中 `yudao-module-mongo` 模块已有代码的审查，我们发现了以下几个与设计指标不符的**严重缺陷（Gaps）**，需要在后续阶段中进行调整和修复：

### 5.1 检索机制：使用模糊正则匹配代替了 Text Index 全文索引
* **当前实现 (`MongoNewsServiceImpl.java:83-88`)**：
  ```java
  Criteria criteria = new Criteria().orOperator(
      Criteria.where("title").regex(searchKey, "i"),
      Criteria.where("keywords").regex(searchKey, "i"),
      Criteria.where("content").regex(searchKey, "i")
  );
  query.addCriteria(criteria);
  ```
* **存在问题**：
  * 这属于**传统的正则表达式子串模糊查询**，并非 MongoDB 的 **Text Index 全文检索**。
  * 该方法**无法支持字段权重配置**（例如标题的权重大于正文），**无法计算相关度得分（textScore）并排序**，且在数据量大时性能极差（会导致全表扫描，无法利用全文索引树）。
* **整改建议**：
  * 在实体类中定义全文索引，并使用 Spring Data MongoDB 的 `TextQuery` / `TextCriteria` 构建查询，例如：
    ```java
    TextCriteria criteria = TextCriteria.forDefaultLanguage().matchingAny(searchKey);
    TextQuery query = TextQuery.queryText(criteria).sortByScore();
    ```

### 5.2 实体类：缺少文本索引注解与定义
* **当前实现 (`MongoNewsDO.java`)**：
  * 实体类没有任何全文索引的标识，仅仅是一个普通的 Spring Data `@Document`。
* **存在问题**：
  * 没有指定 `@TextIndexed`，MongoDB 在启动或初始化时不会自动为 `title`、`content` 和 `keywords` 创建复合全文索引。
* **整改建议**：
  * 修改 `MongoNewsDO.java`，为需要搜索的字段添加 `@TextIndexed` 注解，并指定权重。例如：
    ```java
    @TextIndexed(weight = 10)
    private String title;
    
    @TextIndexed(weight = 5)
    private List<String> keywords;
    
    @TextIndexed(weight = 1)
    private String content;
    ```
  * 同时，可以在 DO 实体中定义一个用来接收分数的字段：
    ```java
    @TextScore
    private Float score;
    ```

### 5.3 网关配置：缺失路由映射规则
* **当前实现 (`yudao-gateway/src/main/resources/application.yaml`)**：
  * 整个网关配置文件中没有关于 `/admin-api/mongo/**` 路径的路由定义，也没有对应的 `grayLb://mongo-server` 目标。
* **存在问题**：
  * 虽然本地开发前端直接配置为 `VITE_BASE_URL='http://localhost:18080'`（直连 monolith 服务端口），绕过了网关，但在**微服务部署、联调环境以及生产环境**中，客户端所有请求均必须经过网关（48080 端口），网关缺失规则将导致访问 `/admin-api/mongo/news/**` 报 `404 Not Found`。
* **整改建议**：
  * 在网关配置文件中，新增 `mongo-server` 的路由配置，例如：
    ```yaml
    - id: mongo-admin-api
      uri: grayLb://system-server # 如果打包在 monolith 中，则路由到 system-server (对应 yudao-server)
      predicates:
        - Path=/admin-api/mongo/**
      filters:
        - RewritePath=/admin-api/mongo/v3/api-docs, /v3/api-docs
    ```

### 5.4 高亮逻辑：当前属于前端弱替换，偏离后端高亮与片段截取要求
* **当前实现**：
  * **后端实现**：在 Java 服务层（[MongoNewsServiceImpl.java](file:///d:/Projects/yudao-cloud/yudao-module-mongo/yudao-module-mongo-server/src/main/java/cn/iocoder/yudao/module/mongo/service/news/MongoNewsServiceImpl.java)）和控制器层，完全**没有任何**文本高亮处理或摘要片段截取逻辑，直接返回完整的原始文本。
  * **前端实现**：在 [index.vue](file:///d:/Projects/yudao-cloud/yudao-ui/yudao-ui-admin-vue3/src/views/mongo/news/index.vue#L241-L252) 中利用 JavaScript 的 `RegExp` 做字符串全局替换，将匹配的 `searchKey` 包装成带样式的 `<mark>` 标签，并通过 `v-html` 动态渲染。
* **存在问题与优化必要性分析**：
  1. **严重偏离设计要求**：`workflow.md` 中阶段四明确要求“*实现高亮：通过 MongoDB 聚合管道 $search 或 $meta + 自行截取（需调研），返回高亮片段*”，即高亮与摘要片段提取应在**后端 Service 层**完成。
  2. **多关键词检索时的“高亮失效” Bug**：
     * 如果用户输入多个关键词以空格分隔（如 `"物联网 创新"`），底层的全文检索（如果修复为真正的 Text Query）能够对这两个词进行“或”或“且”查询，并返回命中其中任意一个词的文档。
     * 然而，前端的高亮逻辑只拿整串 `"物联网 创新"` 作为 RegExp。由于正文里很少会有刚好连在一起的 `"物联网 创新"` 六个字，这会导致前端正则匹配失败，**最终在 UI 界面上无法高亮出任何匹配词**。
  3. **超长文本的“带宽与渲染双重性能灾难”**：
     * 全文检索中，单个文档正文可能长达数万字。目前前端高亮必须拉取整个 `content` 全文，导致网络传输消耗大、加载迟缓。
     * 浏览器在渲染数万字带有多处 `<mark>` 标签的 `v-html` 时，会产生大量的 DOM 重绘，导致 UI 界面严重卡顿甚至崩溃。
  4. **XSS 跨站脚本安全风险**：
     * 前端使用 `v-html` 来解析并渲染文本。由于企业对安全合规及涉密保护要求高，若文档内容中含有恶意脚本，极易引发 XSS 漏洞。
* **整改与优化建议**：
  * **核心改造方案**：将高亮与片段提取逻辑**下沉至后端服务层**。
  * **具体 Java 实现步骤**：
    1. 在 Service 层接收到查询请求后，提取分词列表（例如，支持空格分隔解析为 `List<String> keywords`）。
    2. 执行查询获得文档列表后，遍历每个文档，定位关键词在 `content` 中首次出现的位置（Offset）。
    3. 自行截取该位置前后一定长度（如前后各 80-100 字符）的字符串作为 **“摘要片段（Snippet）”**，并在片段及 `title` 中，使用 Java 字符串替换（或正则替换）将关键词包裹上 `<mark>` 标签（如利用 `String.replace()` 或 Hutool `StrUtil`）。
    4. 定义一个新的传输实体 `MongoNewsRespVO`，后端将只返回高亮处理后的标题 `titleHighlight` 以及摘要 `contentSnippetHighlight`（无需传输完整的几万字正文给分页列表）。
    5. 前端列表组件只负责展示 `titleHighlight` 与 `contentSnippetHighlight`，仅在用户点击“详情”时，再按需请求获取单条文档的全文，从根本上解决带宽和渲染性能瓶颈。
    6. 前端渲染 `v-html` 前，后端应对正文内容进行 HTML 转义（Sanitization），确保只保留合法的 `<mark>` 标签，阻断 XSS 风险。
