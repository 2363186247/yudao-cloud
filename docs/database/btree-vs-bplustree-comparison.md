# B 树 (B-Tree) 与 B+ 树 (B+ Tree) 底层节点存储结构与检索原理规范

本文档对比关系型数据库（MySQL InnoDB）与文档型数据库（MongoDB WiredTiger）在索引数据结构选用（B+ 树 vs B 树）上的底层存储差异与检索机制。

---

## 1. 节点定义与存储模型

以数据库员工表 `employee`（`id` 为索引键 Key，`name, age, address` 为行数据 Data）为例：

| id (主键 Key) | name (姓名) | age (年龄) | address (地址/大文本 Data) |
| :--- | :--- | :--- | :--- |
| **10** | 张三 | 25 | 北京市朝阳区... |
| **20** | 李四 | 30 | 上海市浦东区... |
| **30** | 王五 | 28 | 深圳市南山区... |
| **40** | 赵六 | 35 | 杭州市西湖区... |
| **50** | 钱七 | 22 | 广州市天河区... |

### 1.1 节点职责划分
*   **非叶子节点 (Non-Leaf Nodes)**：树结构的中间路由节点，存储索引键及子节点指针，充当定位下钻的索引路标。
*   **叶子节点 (Leaf Nodes)**：树结构的最底层终止节点，存储数据物理地址或完整记录行。

---

## 2. B 树 (B-Tree) 存储结构与检索机制

在 **B 树** 结构中，非叶子节点与叶子节点**均直接存储完整的数据记录（Data）**。

```text
                       【非叶子节点 (根节点)】
            [P1 | Key=30, Data=(王五,28,深圳) | P2]
                         /                 \
                        /                   \
                       v                     v
           【叶子节点 1】                   【叶子节点 2】
[Key=10, Data=(张三,25,北京)]    [Key=40, Data=(赵六,35,杭州)]
[Key=20, Data=(李四,30,上海)]    [Key=50, Data=(钱七,22,广州)]
```

*   **单点精准匹配 (`id = 30`)**：访问根节点即命中 `Key = 30`，直接读取根节点内存/磁盘页中的数据返回，耗时为 **1 次 I/O**（最佳时间复杂度 $O(1)$）。
*   **局限性**：非叶子节点包含变长 Data 导致单个磁盘页容纳 Key 数量较少，树扇出率（Fan-out）较低，增加数层高度；且叶子节点间无指针链接，范围查询需要向上回溯下钻。

---

## 3. B+ 树 (B+ Tree) 存储结构与检索机制

在 **B+ 树** 结构中：
1.  **非叶子节点**：**仅存储 Key 与子节点指针**，绝不存储任何业务 Data。
2.  **叶子节点**：存储全量 Key 与业务 Data，且所有叶子节点按 Key 字典序组成**双向链表**。

```text
                           【非叶子节点 (根节点)】
                         [P1 | Key=30 | P2]
                               /        \
                              /          \
                             v            v
                 【叶子节点 1】 ⬅️ 链表 ➡️ 【叶子节点 2】
      [Key=10, Data=(张三,25,北京)]     [Key=30, Data=(王五,28,深圳)]
      [Key=20, Data=(李四,30,上海)]     [Key=40, Data=(赵六,35,杭州)]
                                        [Key=50, Data=(钱七,22,广州)]
```

*   **精准匹配 (`id = 30`)**：根节点比对后下钻至叶子节点 2 命中数据，固定消耗 **3~4 次 I/O**。
*   **范围查询 (`WHERE id BETWEEN 18 AND 45`)**：二分定位到 `id = 20` 所在的叶子节点 1 后，**直接沿双向链表向右顺序扫描**至叶子节点 2，无需回溯根节点，随机 I/O 开销极低。

---

## 4. 字符串在 B+ 树中的匹配机制与最左前缀原则

字符串依据字符编码（ASCII/UTF-8）的**字典序（Lexicographical Order）**升序排列。

*   **等值匹配 (`title = 'Nacos'`)**：沿 B+ 树分支下钻至叶子节点精准查找。
*   **前缀匹配 (`title LIKE 'Nacos%'`)**：定位至首个以 `"Nacos"` 开头的叶子节点条目，沿双向链表向右顺序读取，直至前缀不匹配时终止，有效利用索引。
*   **前置通配符匹配 (`title LIKE '%Nacos%'`)**：前置字符不确定导致破坏字典序与最左前缀原则（Leftmost Prefix Rule），无法判定分支下钻方向，被迫退化为叶子节点链表的全表扫描（Full Table Scan）。

---

## 5. MongoDB WiredTiger 普通字段选用 B 树的技术考量

| 对比维度 | B 树 (B-Tree) | B+ 树 (B+ Tree) |
| :--- | :--- | :--- |
| **非叶子节点数据** | 包含 Key 与物理 Data | 仅包含 Key 与指针 |
| **单点 Lookup 性能** | **热点数据在非叶子节点 1~2 次 I/O 直接命中** ($O(1)$) | 必须硬性下钻至底层叶子节点 (3~4 次 I/O) |
| **数据模型契合度** | 适合 Schema-less 动态变长 BSON 文档 | 适合固定 16KB 页面的规范化 SQL 行记录 |
| **典型应用引擎** | **MongoDB WiredTiger 普通字段索引** | **MySQL InnoDB 存储引擎** |

MongoDB 80% 以上的操作为单文档 Lookup（`findOne({_id: xx})`），且提倡反范式嵌入（减少跨表 Join 和范围 Scan 需求），因此普通字段索引权衡选择了单点查询更优秀的 B 树。
